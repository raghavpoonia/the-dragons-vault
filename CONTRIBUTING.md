# Contributing to The Dragon's Vault

First off, thank you for considering contributing to The Dragon's Vault! It's people like you who make this a valuable resource for the security community.

## üéØ Types of Contributions

### 1. Bug Fixes
- Typos and grammatical errors
- Broken links
- Formatting issues
- Code/YAML syntax errors

### 2. Content Improvements
- Clarifying confusing sections
- Adding better examples
- Updating outdated information
- Improving structure and flow

### 3. New Content
- Additional case studies
- New frameworks or methodologies
- Tool comparisons and reviews
- Detection rules and playbooks

### 4. Translations
- Translating chapters to other languages
- Maintaining translation accuracy

## üìù How to Contribute

### Quick Fixes (Typos, Small Changes)
1. Click "Edit" on the file in GitHub
2. Make your changes
3. Submit a pull request with clear description

### Larger Contributions
1. **Fork** the repository
2. **Create a branch**: `git checkout -b feature/your-feature-name`
3. **Make changes** following style guide below
4. **Test** that YAML is valid
5. **Commit**: `git commit -m "Add: brief description"`
6. **Push**: `git push origin feature/your-feature-name`
7. **Submit PR** with detailed description

## üìê Style Guide

### YAML Format
```yaml
section_name:
  
  subsection:
    
    concept:
      what: "Brief definition"
      
      why_it_matters: "Explanation of importance"
      
      example: |
        Multi-line example
        with proper indentation
      
      common_mistakes:
        - "First mistake and how to avoid it"
        - "Second mistake"
      
      key_takeaway: "One-sentence summary"
```

### Writing Style
- **Conversational but precise**: Write like talking to a colleague, not academic paper
- **Practitioner-focused**: Real examples, not just theory
- **Actionable**: Include what to do, not just what to know
- **Honest**: Include failures and mistakes, not just successes
- **Structured**: Use consistent formatting and hierarchy

### Content Guidelines
- **Battle-tested**: Share what you've actually done, not what you read
- **Balanced**: Include pros/cons, trade-offs, context
- **Attributed**: Credit sources, tools, people who taught you
- **Timeless**: Focus on principles over specific tool versions
- **Accessible**: Explain jargon, define acronyms first use

## ‚úÖ Pull Request Process

1. **Title**: Clear, descriptive (e.g., "Add: False positive reduction techniques")
2. **Description**: Explain what and why
3. **Testing**: Confirm YAML is valid
4. **Review**: Respond to feedback constructively
5. **Patience**: Reviews may take a few days

## üö´ What We Don't Accept

- Marketing content or product promotion
- Plagiarized content
- Malicious code or security vulnerabilities
- Content that doesn't add value
- Personal attacks or unprofessional language

## üìú Code of Conduct

- Be respectful and professional
- Welcome newcomers and different perspectives
- Give constructive feedback
- Focus on what's best for the community
- No harassment, discrimination, or toxicity

## üêõ Reporting Issues

Found a problem? Open an issue with:
- **Clear title**: "Section 5.1.12: Incorrect example for delegation"
- **Description**: What's wrong and why
- **Suggestion**: How to fix it (if you have one)
- **Location**: Specific file and line number

## üí° Suggesting New Content

Want to see something covered? Open an issue with:
- **Topic**: What should be covered
- **Why it matters**: Why this is valuable
- **Your experience**: Have you dealt with this?
- **Willing to write**: Can you contribute content?

- Credited in release notes
